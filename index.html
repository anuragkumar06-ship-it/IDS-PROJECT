Section 01

Introduction

Data structures are foundational to computer science — they determine how information is stored, organised, and accessed. Among hierarchical structures, the Binary Search Tree (BST) holds a special place: it combines the structural elegance of a binary tree with the ordering discipline of binary search, enabling efficient lookups, insertions, and deletions in O(log n) time for balanced trees.

A key operation on any tree is traversal — visiting each node systematically, exactly once. Unlike linear data structures where traversal is straightforward, trees branch in multiple directions, necessitating deliberate algorithms that define the order of visits. The choice of traversal order directly impacts what can be achieved: sorting, copying, evaluating, or deleting a tree.

This project implements and analyses the three classical depth-first traversal strategies on a BST:

  • Pre-order (Root → Left → Right) — visits the current node before its children
  • In-order (Left → Root → Right) — visits the current node between its children
  • Post-order (Left → Right → Root) — visits the current node after its children
📌

Project Objective: Implement In-order, Pre-order, and Post-order traversals of a Binary Search Tree — demonstrating a clear understanding of recursive logic, tree node structure, and the practical significance of traversal ordering in real-world computing.

The implementation is written in Python 3 using an object-oriented approach. All algorithms are recursive and original. The report is structured as follows: we first establish the theoretical background of BSTs, then explain the logic and execution of each traversal with step-by-step traces, present the complete code, analyse complexity, and identify applications.

Section 02

Binary Search Tree — Background

What is a Binary Tree?

A binary tree is a hierarchical, non-linear data structure in which each node has at most two children — a left child and a right child. Unlike arrays or linked lists, binary trees enable branching paths through data, making them well-suited for hierarchical relationships and recursive algorithms.

The BST Ordering Property

A Binary Search Tree is a binary tree with a strict ordering constraint imposed on every node:

⚖️

BST Ordering Property: For every node N — all values in N's left subtree are strictly less than N.data, and all values in N's right subtree are strictly greater than N.data. This property must hold recursively for every node in the tree.

🌱

Root Node

The topmost node with no parent. Every traversal originates here.

🍃

Leaf Node

A node with no children (both left and right are None). The recursion base case.

📐

Height

Length of the longest root-to-leaf path. Governs worst-case traversal space (call stack depth).

◀

Left Subtree

All nodes with values strictly less than the current node's value.

▶

Right Subtree

All nodes with values strictly greater than the current node's value.

🔁

Recursive Structure

Every subtree of a BST is itself a valid BST — the fundamental property enabling clean recursion.

Section 03

Logic of Traversal Operations

All three traversal algorithms are depth-first — they explore one branch as deep as possible before backtracking. They differ only in when the current node (Root) is processed relative to its left (L) and right (R) subtrees. Each algorithm is naturally expressed using recursion, leveraging the recursive structure of the BST itself.

Pre-order Traversal

Root → Left → Right

The current node is visited before (pre) either subtree. Process the node first, then recurse left, then recurse right.

Root → L → R
In-order Traversal

Left → Root → Right

The current node is visited between (in) the two subtrees. Recurse left, process the node, recurse right. Yields sorted output on a BST.

L → Root → R
Post-order Traversal

Left → Right → Root

The current node is visited after (post) both subtrees are fully processed. Recurse left, recurse right, then process the node.

L → R → Root
💡

Critical Insight: In-order traversal of any BST always yields all elements in ascending sorted order. This is the defining property of the BST ordering invariant manifested through traversal — effectively a free O(n) sort of any BST's elements.

    Section 04

    Pre-order Traversal — Root → L → R

    In Pre-order traversal, the current node is visited before either subtree is explored. This "visit before descend" approach means the root is always the first node visited, and children are always visited after their parent.

    Pre-order Algorithm — Root → Left → Right
    1. Base case: If the current node is None, return immediately. This is the stopping condition for recursion.
    2. Visit Root: Process (print/store) the current node's data.
    3. Recurse Left: Recursively apply Pre-order to the left subtree.
    4. Recurse Right: Recursively apply Pre-order to the right subtree.

    Step-by-step Execution Trace

    Execution Trace on BST (root = 50)
    Step 150→Visit root node 50
    Step 230→Go left from 50; visit node 30
    Step 320→Go left from 30; visit node 20 (leaf)
    Step 440→Backtrack to 30; go right; visit node 40 (leaf)
    Step 570→Backtrack to 50; go right; visit node 70
    Step 660→Go left from 70; visit node 60 (leaf)
    Step 780→Backtrack to 70; go right; visit node 80 (leaf)
    Output 50→ 30→ 20→ 40→ 70→ 60→ 80
    🌳

    When to use Pre-order: Use pre-order when a parent node must be processed before its children — such as copying or cloning a tree, serialising a tree structure to disk or network, or printing directory listings (folder name before files). The root always appears first in the output.

    Section 05

    In-order Traversal — L → Root → R

    In In-order traversal, the algorithm fully explores the left subtree before visiting the current node, then explores the right subtree. For a BST, this produces all elements in ascending sorted order — making it the most important traversal for sorted data retrieval.

    In-order Algorithm — Left → Root → Right
    1. Base case: If the current node is None, return immediately.
    2. Recurse Left: Recursively apply In-order to the left subtree. Go as deep left as possible first.
    3. Visit Root: Process (print/store) the current node's data.
    4. Recurse Right: Recursively apply In-order to the right subtree.

    Step-by-step Execution Trace

    Execution Trace on BST (root = 50)
    Step 120→Descend left from 50→30→20; 20 is leftmost leaf — visit it
    Step 230→Backtrack to 30; left subtree done — visit node 30
    Step 340→Go right from 30; visit leaf 40
    Step 450→Backtrack to root 50; entire left subtree done — visit root 50
    Step 560→Go right from 50 to 70; descend left to 60 — visit leaf 60
    Step 670→Backtrack to 70; left subtree done — visit node 70
    Step 780→Go right from 70; visit leaf 80 (rightmost)
    Output 20→ 30→ 40→ 50→ 60→ 70→ 80
    ✅

    The Sorted Output Property: The output 20 → 30 → 40 → 50 → 60 → 70 → 80 is in perfect ascending order. This is not a coincidence — it directly proves the BST ordering property. In-order traversal is used as the foundation for BST-based sorting, database range queries, and sorted iteration of sets and maps.

    Section 06

    Post-order Traversal — L → R → Root

    In Post-order traversal, both subtrees are fully explored before the current node is visited. The root is always visited last. This "children before parent" ordering is essential for operations where a parent cannot be processed until its children have been handled.

    Post-order Algorithm — Left → Right → Root
    1. Base case: If the current node is None, return immediately.
    2. Recurse Left: Recursively apply Post-order to the left subtree entirely.
    3. Recurse Right: Recursively apply Post-order to the right subtree entirely.
    4. Visit Root: Only after both subtrees are complete, process (print/store) the current node's data.

    Step-by-step Execution Trace

    Execution Trace on BST (root = 50)
    Step 120→Descend to leftmost leaf 20; both children are None — visit 20
    Step 240→Backtrack to 30; go right to leaf 40; both children None — visit 40
    Step 330→Both subtrees of 30 complete — now visit node 30
    Step 460→Go right from 50 to 70; descend left to leaf 60 — visit 60
    Step 580→Backtrack to 70; go right to leaf 80 — visit 80
    Step 670→Both subtrees of 70 complete — now visit node 70
    Step 750→Both subtrees of root 50 fully processed — visit root last
    Output 20→ 40→ 30→ 60→ 80→ 70→ 50
    🗑️

    When to use Post-order: Post-order is essential when a node cannot be processed until its children are handled. Prime examples include deleting a tree from memory safely (children freed before parent, preventing dangling pointers), evaluating expression trees (operands computed before operators), and compiler code generation from abstract syntax trees.

# ── Node Class ───────────────────────────────────────────────────────────────
class Node:
    """
    Represents a single node in a Binary Search Tree.

    Attributes:
        data  (int)  : The integer value stored in this node.
        left  (Node) : Reference to the left child node.  None if absent.
        right (Node) : Reference to the right child node. None if absent.
    """
    def __init__(self, data):
        self.data  = data
        self.left  = None
        self.right = None


# ── Binary Search Tree Class ──────────────────────────────────────────────────
class BinarySearchTree:
    """
    Binary Search Tree with insert and three DFS traversal methods.

    BST Property: For every node N —
        All values in N's left subtree  < N.data
        All values in N's right subtree > N.data
    """

    def __init__(self):
        self.root = None

    # ── Insertion ─────────────────────────────────────────────────────────────
    def insert(self, data):
        """Insert a value into the BST, maintaining the BST property."""
        if self.root is None:
            self.root = Node(data)           # Tree is empty; new node becomes root
        else:
            self._insert_rec(self.root, data)

    def _insert_rec(self, current, data):
        """Recursive helper: navigate to the correct position and insert."""
        if data < current.data:            # Go left for smaller values
            if current.left is None:
                current.left = Node(data)
            else:
                self._insert_rec(current.left, data)
        elif data > current.data:          # Go right for larger values
            if current.right is None:
                current.right = Node(data)
            else:
                self._insert_rec(current.right, data)
        # Duplicates are silently ignored

    # ── Pre-order: Root → Left → Right ────────────────────────────────────────
    def preorder(self):
        """
        Pre-order traversal: Root → Left → Right.
        Returns a list of node values in pre-order sequence.
        Use case: tree cloning, serialisation, directory listing.
        """
        result = []
        self._preorder_rec(self.root, result)
        return result

    def _preorder_rec(self, node, result):
        if node is None:                   # Base case
            return
        result.append(node.data)           # Step 1 — Visit Root
        self._preorder_rec(node.left,  result)  # Step 2 — Traverse Left
        self._preorder_rec(node.right, result)  # Step 3 — Traverse Right

    # ── In-order: Left → Root → Right ─────────────────────────────────────────
    def inorder(self):
        """
        In-order traversal: Left → Root → Right.
        Returns a list of node values in in-order sequence.
        Key property: produces values in ASCENDING SORTED ORDER for any BST.
        Use case: sorting, range queries, database indexing.
        """
        result = []
        self._inorder_rec(self.root, result)
        return result

    def _inorder_rec(self, node, result):
        if node is None:                   # Base case
            return
        self._inorder_rec(node.left,  result)  # Step 1 — Traverse Left
        result.append(node.data)               # Step 2 — Visit Root
        self._inorder_rec(node.right, result)  # Step 3 — Traverse Right

    # ── Post-order: Left → Right → Root ───────────────────────────────────────
    def postorder(self):
        """
        Post-order traversal: Left → Right → Root.
        Returns a list of node values in post-order sequence.
        Root is always the LAST element in the output.
        Use case: tree deletion, expression evaluation, code generation.
        """
        result = []
        self._postorder_rec(self.root, result)
        return result

    def _postorder_rec(self, node, result):
        if node is None:                    # Base case
            return
        self._postorder_rec(node.left,  result)  # Step 1 — Traverse Left
        self._postorder_rec(node.right, result)  # Step 2 — Traverse Right
        result.append(node.data)                 # Step 3 — Visit Root


# ── Driver Code ───────────────────────────────────────────────────────────────
if __name__ == "__main__":

    bst = BinarySearchTree()

    # Build the BST by inserting values in this order
    values = [50, 30, 70, 20, 40, 60, 80]
    for val in values:
        bst.insert(val)

    # Run and display all three traversals
    pre  = bst.preorder()
    ino  = bst.inorder()
    post = bst.postorder()

    print("=" * 50)
    print("  BST Traversal Results")
    print("=" * 50)
    print(f"  Pre-order  (Root→L→R) : {pre}")
    print(f"  In-order   (L→Root→R) : {ino}")
    print(f"  Post-order (L→R→Root) : {post}")
    print("=" * 50)
    print(f"  ✓ In-order is sorted: {ino == sorted(values)}")
    print("=" * 50)

    Section 08

    Time & Space Complexity Analysis

    All three traversal algorithms visit every node in the BST exactly once. No node is visited twice; no node is skipped. This immediately establishes that all traversals run in O(n) time, where n is the total number of nodes.

    Operation Time Complexity Space (Call Stack) Best Case Worst Case
    Pre-order O(n) O(h) O(log n) O(n)
    In-order O(n) O(h) O(log n) O(n)
    Post-order O(n) O(h) O(log n) O(n)
    BST Insert (single) O(log n) O(1) O(log n) O(n)
    BST Search O(log n) O(1) O(1) O(n)

    Where h denotes the height of the BST:

    • Balanced BST: h = ⌊log₂(n)⌋ → Space = O(log n). For our 7-node tree, h = 2, so the recursion stack depth is at most 3 frames.
    • Degenerate (skewed) BST: Inserting already-sorted data (e.g., 10, 20, 30, 40…) creates a "right chain" where h = n − 1. Space degrades to O(n) and the BST loses its efficiency advantage. AVL Trees or Red-Black Trees solve this via auto-balancing.
    ⚠️

    Space Note on Recursion: Each recursive call consumes one stack frame. The maximum number of simultaneous stack frames equals the height h. For our balanced 7-node BST, this is 3 frames — very efficient. In production systems with millions of nodes, iterative traversal using an explicit stack is preferred to avoid stack overflow risks.

    Section 09

    Real-world Applications

    BST traversals are not abstract academic exercises — they are foundational to real-world systems across databases, operating systems, compilers, and artificial intelligence. The choice of traversal order directly determines what can be achieved.

    📁

    File System Traversal Pre

    Operating systems represent directories as trees. Pre-order lists parent folders before their contents — used in ls -R, find, and file explorers. The parent must appear before its children.

    🗃️

    Database Index Range Queries In

    SQL engines use B+ trees for indexes. In-order traversal enables efficient range scans — finding all records with keys between a and b in sorted order. The backbone of SQL WHERE key BETWEEN a AND b.

    📤

    Tree Serialisation & Cloning Pre

    Pre-order output uniquely encodes a BST structure. The same tree can be reconstructed from this sequence. Used to persist BSTs to disk, send over a network, or deep-copy a tree data structure.

    🧮

    Expression Tree Evaluation Post

    Arithmetic expressions are stored as trees (operators at internal nodes, operands at leaves). Post-order evaluates operands before operators — the natural evaluation order for stack-based calculators and compilers.

    🔢

    BST Sort Algorithm In

    Insert n elements into a BST (O(n log n)), then in-order traverse (O(n)) to retrieve them sorted. This is the basis of Tree Sort, a comparison-based sorting algorithm with the same average complexity as heapsort.

    🗑️

    Safe Memory Deallocation Post

    In C/C++, freeing a tree requires freeing children before the parent node. Post-order ensures no dangling pointers are created. Garbage collectors in managed languages use similar bottom-up approaches.

    🌳

    Abstract Syntax Trees (Compilers) Pre

    Compilers parse source code into ASTs. Pre-order traversal generates intermediate code top-down; post-order generates code bottom-up (children evaluated before parent operations). Different compiler phases use different traversals.

    🤖

    AI Decision Trees InPost

    Machine learning decision trees use in-order traversal to extract sorted feature thresholds, and post-order for bottom-up pruning — removing leaf nodes that don't improve accuracy, starting from leaves and working up.

    Section 10

    Conclusion

    This project investigated the Binary Search Tree — a hierarchical data structure that enforces a strict ordering property enabling efficient search, insertion, and sorted traversal in O(log n) time for balanced trees. We designed, traced, and implemented all three canonical depth-first traversal strategies.

    The key insight is that the same recursive structure of the BST underpins all three algorithms — only the moment of processing changes. In-order yields sorted output by always visiting left children first; Pre-order clones or serialises by visiting the root first; Post-order safely deletes or evaluates by visiting the root last.

    All three algorithms are O(n) in time and O(h) in space, where h is the tree height. Their simplicity, power, and wide applicability make them foundational knowledge for any computer science student.
